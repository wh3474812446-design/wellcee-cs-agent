# 编排框架教学：从 Dify 到纯代码实现

> 2026-07-09 ｜ 教学文档（配套可运行代码：`standalone/pipeline.py`）
> 三个问题一次讲透：① 我们的 Dify 编排本质是什么框架 ② 每个环节的提示词为什么那么写 ③ 不用 Dify 用代码怎么做、Agent 框架怎么选

---

## 第一课：剥掉 Dify 的皮——我们用的编排框架本质

### 1.1 它就是一个「四层 DAG 状态机」

把 19 个节点的画布抽象掉，结构只有四层：

```
用户输入
  ↓
【确定性预处理层】 语言检测(代码) → 高风险词规则(if-else)     ← 不花钱、零延迟、100% 可预测
  ↓
【路由层】         意图分类(LLM 判别，6 类)                  ← 唯一的"判断型"LLM 调用
  ↓
【处置层】         五种处置模板 + 一个静态回答：
                   faq: 检索→受限生成        guide: 检索→分步引导(带记忆)
                   tool: 参数提取→API→组织回复  handoff: 状态收集(带记忆)→工单摘要
                   chitchat: 人设应答         unclear: 固定文案(不走LLM)
  ↓
输出（流式）
```

**为什么是这个形状**——三条设计公理，比框架本身重要：

| 公理 | 含义 | 在我们编排里的体现 |
|------|------|------------------|
| **规则优先于模型** | 确定性手段能覆盖的，绝不交给概率性手段 | 语言检测用字符占比不用 LLM；「举报/被骗/报警」强信号词直接旁路，不赌分类器 |
| **检索限定生成**（Grounded Generation） | 生成的每个事实必须有出处，检索结果就是生成的"围栏" | FAQ 节点 prompt 第一句就是"仅基于下方检索结果回答"；检索为空→诚实兜底而非自由发挥 |
| **红线随身携带** | Chatflow 没有全局 system，每个 LLM 节点是独立调用——红线必须注入每一个出口 | 公共人设前缀（含 6 条红线）贴在全部 5 个 LLM 节点的 SYSTEM 开头，漏一个节点=漏一个幻觉出口 |

### 1.2 这属于哪一类架构：Workflow，不是 Agent

业界通用区分（Anthropic《Building Effective Agents》的框架）：
- **Workflow**：LLM 和工具按**预定义的代码路径**编排——每一步走哪条分支是人设计好的
- **Agent**：LLM **自主决定**下一步做什么、调用什么工具、何时结束

我们的客服系统是**纯 Workflow**。这是刻意选择而非能力不足：客服场景要的是**可预测、可评测、可审计**——每条消息流经哪些节点有完整路径记录（评测归因就靠它），错了能定位到具体节点。如果做成自主 Agent（模型自己决定查不查知识库、转不转人工），评测和红线控制的难度会上一个数量级，而收益（灵活性）在客服场景几乎用不到。

**面试金句**：能用 Workflow 解决的问题不要上 Agent——自主性是成本不是卖点，只有当任务的步骤无法预先枚举时（如开放式研究、复杂排障），才值得为 Agent 的不可预测性买单。

## 第二课：提示词解剖——每个环节的 SYSTEM 及其设计原理

所有 prompt 共用一个骨架，先看骨架再看各节点：

```
[公共人设前缀]     ← 我是谁、什么语气、什么语言、6 条红线（每个节点都带）
【本节点任务】     ← 一句话定义此节点唯一职责（单一职责原则）
规则：            ← 有序列表，越靠前优先级越高（模型对列表前部的遵循度更高）
- 条件行为（若X则Y）
- 禁止项（显式写"禁止/不要"）
- 输出格式约束
[变量注入]        ← {{#检索结果#}} / {{#查询结果#}}，数据与指令分离
```

### 2.1 公共人设前缀（贴在每个 LLM 节点最前）

```
你是小Cee，Wellcee（唯心所寓）的 AI 智能客服助手。Wellcee 是无中介费的房东直租+社交租房平台，覆盖 40+ 城市、用户来自 170+ 国家。

【人设】年轻、友好、专业、有温度；适度使用 emoji（每条不超过 2 个）；称呼用户"你"。

【语言】lang={{#语言检测.lang#}}。zh → 简体中文回答；en → 英文回答。用户中英夹杂时跟随其主要语言。

【红线（任何情况下不可违反）】
1. 金额、押金、退款时效、法律责任类信息：只能使用知识库/检索结果/查询结果中的原文，禁止编造或自行估算
2. 禁止承诺性/结论性表述："一定、保证、肯定能退、他确实是骗子"等
3. 时效类回答必须带限定语（"以实际到账为准，超时请转人工核查"）
4. 不评价纠纷双方对错，不预测处理结果
5. 被问身份时明示自己是 AI 助手
6. 答不了就诚实说，并给转人工路径；禁止编造
```

**解剖**：
- 人设段给「谁+怎么说话」，emoji 限 2 个是量化约束（"适度"这种词模型执行不稳，数字才稳）
- 语言段用**变量注入**而非让模型自己判断——上游代码节点已经判定了 lang，模型只执行不判断（能确定的不留给概率）
- 红线 6 条全部是**否定式+可验证**的写法：每条都能在评测里客观判定是否违反（这正是评测规范 D 维度的逐条来源——**prompt 红线与评测标准同源**，这是"评测驱动开发"的关键闭环）

### 2.2 意图分类（问题分类器节点，flash，temperature 0.1）

不写自由 prompt，写**6 个类别定义**（Dify 分类器把它们拼进内置模板）：

```
faq_direct：询问平台规则、功能说明、费用、安全性、原因解释等有标准答案的问题。
  例：押金托管规则、怎么联系房东、收不到验证码、平台怎么收费、为什么要认证、
     房东不回复/不理我怎么办、房源没人看/没曝光怎么办、landlord not replying
process_guide：询问"怎么操作、怎么办"的流程类问题，需要分步指导。
  例：线上签约怎么走、押金怎么退、怎么发布房源、学生认证怎么做
tool_deposit：查询自己这笔押金的退还状态或进度。例：我的押金退了吗、帮我查下订单
human_handoff：举报、纠纷、申诉、账号安全（换绑/注销）等必须人工处理的问题。
  注意：单纯的房东不回复消息、房源曝光低属于 faq_direct，不属于本类
chitchat：打招呼、闲聊、询问你是谁/是不是机器人、夸奖或吐槽助手本身
unclear：表述模糊无法判断在问什么，或不属于以上任何类别
```

**解剖**：
- 类别描述 = 定义 + **例句**。例句就是分类器的"少样本训练集"——首轮评测 3 个 badcase（V02/V07/E10）全是例句缺口造成的，修复方式就是补例句+加排除说明（human 类那句"注意：不属于本类"）。**运营意图分类器 = 运营例句库**，这是可以写进周迭代 SOP 的
- temperature 0.1：判别任务要稳定不要创造性
- 6 类而非 20 类（意图表有 20 个 MVP 意图）：**分类粒度=处置粒度**。workflow 只需要知道"怎么处理"，不需要知道"具体哪个意图"——20 个意图的细粒度活在知识库标签和评测集里

### 2.3 FAQ 直答节点（pro + thinking:false）

```
【本节点任务】仅基于下方检索结果回答用户问题。
规则：
- 答案内容必须完全来自检索结果，禁止补充检索结果之外的事实
- 若检索结果为空或与问题无关：诚实告知"这个问题我暂时没有可靠资料"，给出两条路径
 （点击"转人工"/换个说法再问），禁止编造，此时**不要**输出来源行
- 来源标注规则：仅当答案的主体事实来自检索结果时才在末尾标注来源；如果你给的是
  一般性建议或常识（检索结果只是间接相关），不要标来源。标签跟随回答语言——
  中文「📖 来源：<条目问题>」，英文「📖 Source: <entry question>」
- 涉及金额/时效：使用原文表述并附限定语
检索结果：{{#context#}}
```

**解剖**：
- 第一行"仅基于"是整个 RAG 的安全带；「禁止补充检索结果之外的事实」把"仅基于"变成可执行禁令
- 兜底话术给了**固定结构**（承认+两条路径）而不是让模型自由发挥——兜底话术是产品文案，不是模型创作
- 来源标注规则的"仅当主体事实来自检索"是首轮评测 H04 badcase（来源失实）后**收紧**的——prompt 是版本化演进的，不是一次写对的

### 2.4 流程引导节点（pro，带记忆 10 轮）

```
【本节点任务】用户在问操作流程。基于检索结果把流程拆成编号步骤，分步引导，不一次灌完。
规则：
- 第一轮：先确认关键分支条件（例：押金问题先问"是否走了线上签约？"；发布问题先确认
  角色），给对应路径的第 1-2 步，末尾问"继续吗？"
- 用户确认后再给后续步骤
- 每步一句话说清：在哪个页面、点什么、会看到什么
- 检索结果覆盖不了的步骤细节，明说"这一步的具体入口以 App 实际界面为准"，不编造按钮名
检索结果：{{#context#}}
```

**解剖**：
- "先确认分支条件"直接来自对话流程图①的设计（线上/线下签约答案完全不同）——**流程图是 prompt 的规格说明书**
- "每步一句话说清在哪、点什么、看到什么"是把 UX 写作规范塞进 prompt
- "不编造按钮名"是红线 1 在流程场景的具体化——通用红线+场景红线双保险

### 2.5 工具调用-查询回复节点（pro）

```
【本节点任务】根据查询结果回复用户押金状态。
查询结果：found={{#mock押金查询.found#}}，status={{#mock押金查询.status#}}，
         eta={{#mock押金查询.eta#}}，amount={{#mock押金查询.amount#}} 元
规则：
- found=none（用户没给订单号）：请用户提供订单号（在 App「我的-我的签约」可查看），不要瞎猜
- found=no：告知未查到该订单，请核对订单号，或点"转人工"人工核查
- found=yes：报状态；金额、时效必须原样引用查询结果字段；附"以实际到账为准"
- status=退款处理中：加一句"超过 3 个工作日未到账，随时转人工帮你核查"
```

**解剖**：
- 结构化数据（API 返回）以**具名字段**注入而不是扔一段 JSON——模型对具名字段的引用错误率更低
- 三个 found 状态的行为**逐一枚举**——工具调用后的回复是状态机不是自由对话
- "原样引用查询结果字段"：工具结果也是"检索结果"，同样受红线 1 管辖

### 2.6 转人工收集节点（pro，带记忆 10 轮）——最复杂的一个

```
【本节点任务】用户的问题需要人工处理（举报/纠纷/申诉/账号安全/主动要人工）。你负责：
共情 → 收集信息 → 生成工单摘要。你不解决问题本身，也绝不下结论。
必收字段：① 问题类型 ② 对方昵称或房源链接（如涉及他人）③ 订单号（如涉及押金/签约）
        ④ 用户诉求 ⑤ 证据说明（有无聊天记录/转账凭证等截图）
规则：
- 【最高优先级】用户提及被骗/资金损失时，第一轮回复必须同时包含：① 提醒保留转账凭证
  与聊天记录 ② 建议报警。禁止预测追回结果
- 开场先共情一句；只陈述平台会核实处理，禁止"他确实是中介/肯定能退"类判断
- 对照对话历史：已提供的字段不重复问；一次只问 1-2 个缺失字段
- 字段收齐后输出工单摘要并请用户确认：
  ---
  📋 工单摘要（草稿）
  问题类型：／涉及对象/订单：／用户诉求：／证据：／对话要点：（一句话）
  ---
- 用户确认后：告知已提交，人工客服预计 1-2 个工作日内跟进（以实际为准），安抚收尾
```

**解剖**：
- 这是一个**用 prompt 实现的槽位填充状态机**（slot-filling）：必收字段=槽位，"对照历史不重复问"=状态检查，"一次只问 1-2 个"=节奏控制，工单摘要=终态输出。传统对话系统要写状态机代码，LLM 时代一段 prompt 承载——但逻辑本质没变，这是第三课代码版里看得最清楚的
- 「最高优先级」标记+放规则第一条：首轮评测 T03 badcase（资金损失话术漏出）的修复——**规则在列表中的位置就是它的权重**
- "你不解决问题本身"是给节点定负空间：防止模型在纠纷场景热心给出"你可以起诉他"之类的越权建议

### 2.7 闲聊节点（flash，temperature 0.7）与澄清反问（无 LLM）

```
【本节点任务】用户在闲聊或问你身份。按人设简短回应（不超过 2 句），然后自然引导回业务，
例如："有什么租房相关的问题我可以帮你？😊"
被问"你是人工吗/机器人吗"：明确说明自己是 AI 助手小Cee，复杂问题可以随时转人工。
```
- 唯一 temperature 0.7 的节点（闲聊要一点人味）；"不超过 2 句"防话痨；"引导回业务"是客服产品的门面纪律
- 澄清反问是**固定文案回答节点**，不走 LLM：歧义消解要的是确定性选项（1️⃣-5️⃣），不是更多生成——评测里它 0 耗时 0 成本 100% 稳定

## 第三课：不依赖 Dify，纯代码怎么实现

### 3.1 核心认知：Dify 的 19 个节点 ≈ 200 行 Python

配套代码 `standalone/pipeline.py`（可直接运行，依赖仅 `openai` SDK）。节点↔代码映射：

| Dify 节点 | 代码实现 | 行数量级 |
|-----------|---------|---------|
| 开始/回答 | `respond()` 函数入出口 | — |
| 语言检测（代码节点） | `detect_lang()`：中文字符占比 | 3 行 |
| 高风险规则（if-else） | `risk_hit()`：`any(w in q for w in RISK_WORDS)` | 3 行 |
| 问题分类器 | `classify()`：LLM + JSON 输出 + 枚举校验 | 15 行 |
| 知识检索 | `Retriever` 类：字符 bigram 余弦相似度（54 条量级零依赖够用；生产换 embedding，接口不变） | 30 行 |
| 各 LLM 节点 | `chat()` 封装 + 各 handler 函数（prompt 与 Dify 版完全同源） | 每个 10 行 |
| 参数提取器 | `extract_order_id()`：LLM JSON 输出 | 8 行 |
| mock 押金查询（代码节点） | `mock_deposit_query()`：与 Dify 版逐字相同 | 12 行 |
| 记忆（memory 10 轮） | `session["history"]` 列表切片 `[-20:]` | 2 行 |
| 条件分支/连线 | 一个 `dict` 路由表 + if/else | 10 行 |

**看懂这张表就看懂了低代码平台的本质**：Dify 卖的不是"能力"（能力就是这 200 行），是**可视化协作、版本管理界面、日志观测、发布托管**这些工程外围。所以选型问题从来不是"Dify 能不能做到"，而是"这些外围我要不要自己搭"。

### 3.2 代码版的三个教学要点（读 pipeline.py 时重点看）

1. **检索的本质是相似度排序**：54 条知识库根本不需要向量数据库——字符 bigram 余弦 30 行搞定且中文友好；代码里 `Retriever.search()` 的接口签名（query → top_k 条目）与换成 bge/text-embedding-3 后完全一致。**先用最笨的方法跑通，把复杂度留给数据量证明它必要的那天**（与"并发不过度设计"同一个哲学）
   实测已知局限（保留它正是教学目的）：「押金退了多久到账」bigram 命中 0.32 ✅，但「怎么把房子挂上去」MISS——字面无重叠的同义改写（挂上去=发布）只有语义 embedding 能捕捉。两个观察：① 这就是 Dify 版评测里检索单测的价值——先量化词法基线，再决定要不要上 embedding；② MISS 之后系统走的是 FAQ 节点的诚实兜底话术而非编造——**检索失败≠幻觉**，安全性由 prompt 层兜住，检索层只影响解决率。一个 MISS 把分层设计讲透了
2. **LLM 判别任务的结构化输出**：`classify()` 用 `response_format={"type":"json_object"}` + 输出后枚举校验 + 校验失败落 `unclear`——三层防线让概率输出变成工程可靠的路由信号
3. **状态就是一个 dict**：Dify 的"会话变量/记忆"在代码里就是 `session = {"history": [], "lang": ...}`。转人工的槽位收集不需要额外框架——prompt 里"对照历史不重复问"让 LLM 自己做了状态检查（这是 LLM 时代对话系统与传统槽位状态机的最大区别：状态检查从代码转移进了模型）

### 3.3 什么时候这份纯代码版就够了、什么时候要上框架

纯代码（裸 SDK + 自写循环）够用的边界：**流程是固定 DAG + 单 Agent + 状态简单**——我们的客服完全在此边界内。超出边界的信号才考虑框架（见第四课）。

## 第四课：Agent 框架选型——决策树而非排行榜

### 4.1 先问四个问题（顺序重要）

```
Q1 流程能画成固定的图吗？
   能  → Workflow 形态：低代码平台（Dify/Coze/n8n）或代码 DAG（本文的 pipeline.py）
   不能（步骤需模型自主规划）→ Agent 形态：tool-use 循环（裸 SDK）或 LangGraph 类框架
Q2 谁来维护流程？
   运营/业务要看要改 → 可视化平台价值大（Dify 的编排图运营能看懂，这在跨职能协作里值钱）
   纯工程团队 → 代码优先（版本管理/测试/CR 全套软件工程红利）
Q3 需要多 Agent 协作吗（客服×审核×运营工作流互相调用）？
   是 → LangGraph（图状态机，控制流最强）/ AutoGen、CrewAI（对话式协作抽象）
   否 → 别为用不上的抽象付学习成本
Q4 规模与合规到了什么量级？
   （参照《从Demo到生产》阶段3的触发条件：定制需求平台满足不了 / 日会话>5万 / 多Agent）
```

### 4.2 主流选项速查（按抽象层级从高到低）

| 层级 | 代表 | 适合 | 我们场景的评价 |
|------|------|------|---------------|
| 低代码平台 | **Dify**、Coze、n8n | 验证期、业务侧参与维护、需要现成的发布/日志/标注界面 | ✅ 当前阶段最优：一天出 Demo+运营可读+自带观测 |
| 重代码框架 | **LangGraph**（LangChain 系） | 复杂控制流：循环/回退/人工审批中断/多 Agent 图 | 阶段 3 候选：转人工收集若要做成严格状态机、或客服×审核工作流打通时 |
| RAG 专精 | LlamaIndex | 检索管道很深（多路召回/重排/引用溯源）的知识型应用 | 知识库到几千条、要上重排时引入其检索模块即可，不必整框架迁移 |
| 多 Agent 抽象 | AutoGen、CrewAI | 多角色协作实验 | 客服场景用不到，不引入 |
| 裸 SDK + 自写循环 | OpenAI/Anthropic SDK | 逻辑简单固定、要极致可控与极简依赖 | ✅ pipeline.py 即此路线：200 行、零框架锁定、评测工具直接复用 |

### 4.3 选型的三条反模式（面试防坑）

1. **简历驱动选型**：为了"用过 LangGraph"而把固定流程写成图框架——固定 DAG 用 if/else 最诚实
2. **框架能力=系统能力的错觉**：框架解决的是控制流表达，**不解决**意图体系、红线设计、评测标准——我们项目 80% 的产品价值（意图表/红线/评测规范）在任何框架下都原样成立，这也是为什么敢说"迁移成本可控"
3. **过早抽象**：还没有第二个 Agent 就引入多 Agent 框架、还没有复杂控制流就引入图框架——迁移的正确时机是"痛了再迁"，且迁移验收标准现成：50 条评测集全量回归打平

### 4.4 一句话总结

> 编排框架是**容器**，意图体系+提示词+红线+评测是**内容物**。容器随规模换（Dify → 代码 → LangGraph），内容物一次建成处处复用——这就是为什么本项目把 80% 的功夫花在内容物上。

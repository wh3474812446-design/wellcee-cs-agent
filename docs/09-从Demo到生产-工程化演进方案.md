# 从 Demo 到生产：工程化演进方案

> 2026-07-09 ｜ 配套《上线与迭代规划》（那份讲业务怎么灰度放量，本份讲技术架构怎么演进支撑它）
> **立场先行**：Dify Cloud Demo 的定位是**产品验证器**——用最低成本验证意图体系、对话设计、红线机制是否成立（已验证：评测 94%/红线 0）。它不是、也不应该是生产系统。本文回答的问题：**验证成立之后，通往企业真实场景的路怎么走**——每一节都按「Demo 现状 → 生产 Gap → 方案分级 → PM 在其中的职责」展开，因为 PM 的价值不是替工程师做架构，而是定义约束、排优先级、管理演进节奏。

---

## 一、高并发：先量化，再设计（反对为不存在的规模过度设计）

### Demo 现状
Dify Cloud sandbox：共享算力、无 SLA 承诺、知识库 API 有订阅级限流（实测触发过 403）；LLM 依赖 DeepSeek 官方 API 的 RPM/TPM 配额。

### 第一步不是加机器，是算清楚量级（PM 的活）
千万级注册用户 ≠ 千万级并发。粗算：假设 DAU 50 万🔶 × 客服咨询渗透率 1-2%🔶 = 日会话 5,000-10,000 → 按 12 小时活跃期+晚高峰 3 倍系数，**峰值并发会话约 20-50，峰值 QPS 在十位数以内**。客服不是秒杀场景。
**结论**：应用层并发压力很小，真正的瓶颈是 ① LLM 供应商限流 ② 单会话长链路占用。这个测算决定了方案选型不需要"大厂级"架构——**知道不需要什么，和知道需要什么同样重要**。

### 方案分级
| 层 | 方案 | 触发条件 |
|----|------|---------|
| L1（上线即做） | Dify **私有化部署**（开源版，K8s 双副本起步）替代 Cloud；API 网关做限流与排队 | 灰度阶段② 全量前 |
| L2 | LLM 多供应商冗余：DeepSeek 主 + 备用供应商 failover（网关层切换，应用无感） | 供应商单点故障出现过 1 次即做 |
| L3 | 高频 FAQ **语义缓存**：TOP 意图的标准问命中缓存直接返回（不打 LLM） | 日会话 >5,000 或成本优化期 |
| L4 | 队列削峰：高峰期超阈值会话进等待队列 + 前端预期管理话术 | 峰值并发 >100 |

### PM 职责
- 定 **SLA 与降级产品形态**：可用性目标 99.9%；LLM 全不可用时系统不是白屏，而是**降级为关键词 FAQ 检索 + 一键转人工**（体验有损但服务不中断）——降级态的用户体验是产品设计，不是工程细节
- 把并发测算的假设值（DAU/渗透率）列入上线前向数据团队核实的清单

## 二、时延与性能：分场景定预算，不是全链路求快

### Demo 现状
实测链路：语言检测（代码节点，ms 级）→ 意图分类（flash，1-2s）→ 知识检索（百 ms）→ 生成（flash/pro，5-15s）。首 token 2-5s（流式），完整回答 10-20s。PRD 目标：首 token <3s。

### PM 的第一件事：定义分场景时延预算（不是"越快越好"）
| 场景 | 首 token 预算 | 理由 |
|------|-------------|------|
| FAQ 直答 | <3s | 用户对"查答案"的耐心基线 |
| 工具调用（查押金） | <5s | 有"查询中"的心理预期，可加载动画兜住 |
| 转人工收集/工单摘要 | 可放宽到 8s | 用户已决定等人工，此刻质量>速度 |
| 澄清反问 | <1s | 固定文案不走 LLM（Demo 已如此设计） |

### 优化清单（按 ROI 排序）
1. **流式输出**（已有）——首 token 体验的地板
2. **规则前置扩大覆盖**（已有雏形）：高频标准问法直接规则命中跳过分类器，省 1-2s
3. **语义缓存**：TOP 20 标准问命中缓存 <500ms，预计覆盖 30%+ 流量🔶
4. **分类与检索并行**（推测执行）：不等分类结果先发起检索，分类完成后取用——链路省 1s+
5. **prompt 瘦身**：公共前缀当前约 400 token，生产版按节点裁剪（闲聊节点不需要完整红线全文，只需身份条款）
6. **模型分层**（已有）：分类/提取用 flash，生成用 pro——继续往下分：语言检测已经是 0 成本代码节点
- 监控口径：**p95/p99 分位**而非平均值（长尾才是投诉来源）；按节点拆解链路耗时（Dify 日志自带节点耗时，私有化后接入 APM）

## 三、内部系统集成：mock 的 schema 就是 API 需求文档

### Demo 现状
押金查询是 mock（订单尾号规则）；工单是文本摘要；用户身份靠对话询问（"你是租客还是房东？"）。

### 真实集成清单（按依赖优先级）
| 系统 | 集成内容 | 优先级 | 说明 |
|------|---------|--------|------|
| 用户系统 | 登录态透传：AI 直接知道用户身份/角色/当前订单 | P0 | 消灭"你是租客还是房东"这类本不该问的问题——**上下文即体验** |
| 工单系统 | 写入 API（结构化工单）+ 状态回流（用户追问"我的工单怎样了"） | P0 | 过渡方案：格式化邮件/群机器人（PRD 已列） |
| 订单系统 | 押金状态查询（只读） | P1 | mock 的返回字段（found/status/eta/amount）**就是接口契约草案**——Demo 阶段已把需求定义完了 |
| 审核系统 | 举报线索结构化推送 | P1 | 客服 Agent × 内容审核的联动点（JD 方向） |
| IM/App | 客服入口嵌入、消息推送 | P0 | 拍板 B：与 9 宫格并存、AI 置顶 |
| 知识库 CMS | 运营可视化维护界面 + 双语同步校验 | P2 | MVP 期 Owner 手工维护（评审 C2 承诺过） |

### 集成架构原则
`Dify 工具节点（或自研编排）→ API 网关 → 内部服务`，网关统一三件事：**鉴权**（AI 服务专用凭证+用户态透传）、**审计**（每次内部调用留痕：谁的会话、查了什么、返回了什么）、**脱敏**（手机号/证件号在进入 LLM 上下文前打码）。
**数据权限矩阵是 PM 的交付物**：AI 能读什么（订单状态 ✅）、不能读什么（支付账号明细 ❌）、能写什么（工单 ✅）、绝不能写什么（订单操作 ❌——查询与操作分离，AI 有读权无写权是 v1 的安全边界）。

### 故障与安全
- 内部系统超时/异常 → 节点级降级话术（"系统开小差了，我帮你转人工核实"）+ 自动生成工单，**集成失败不能变成对话死胡同**
- 证据截图等敏感附件不进 AI 侧，直接走工单通道（PRD L2 已定）；对话日志留存与脱敏规则过法务（上线依赖清单已列）

## 四、闭环优化：埋点 → 监控 → AB → 模型选型的完整回路

### 4.1 数据埋点（PRD 埋点章节的落地设计）
三层事件模型：
- **会话级**：入口来源 / 用户角色 / 总轮数 / 最终处置（解决-转人工-流失）/ CSAT
- **轮次级**：意图判定结果与置信路径（规则命中 or 分类器）/ 处置类型 / 是否命中知识库及条目 ID / 首 token 与完整耗时 / 👍👎
- **节点级**：各节点耗时与错误（性能归因用）
P0 四件套（意图/处置/转人工原因/满意度）上线即埋——这是数据飞轮的最小燃料；P1（置信度/命中条目/轮数）两周内补齐。**埋点评审是 PM 主持的会**：每个字段回答"哪个决策会用到它"，答不上来的不埋。

### 4.2 在线监控（三层仪表盘 + 分级报警）
| 层 | 指标 | 频率 | 报警 |
|----|------|------|------|
| 业务 | 自动解决率/转人工率/CSAT/工单字段齐全率 | 日报 | 解决率日环比降>10pp |
| 质量 | 幻觉抽检（每日 20 条人工抽样按评测规范打分）/ 分意图命中率 | 日抽样+周报 | 幻觉 ≥1 例当日复盘（0 容忍） |
| 系统 | p95 时延/节点错误率/LLM token 成本/供应商可用性 | 实时 | p95>5s、错误率>1%、日成本超预算 20% |

### 4.3 AB Test（澄清策略就是第一个现成实验）
- **能 A/B 什么**：prompt 版本、模型选择、澄清策略、开场白与快捷问题、来源展示样式
- **第一个实验就在手上**：拍板 A 遗留的验证——澄清反问 1 次 vs 2 次（评审时判断分歧的点，正适合数据说话）。实验设计：分流单元=会话；主指标=澄清后意图命中率与最终解决率；护栏=转人工率、CSAT、会话时长
- **样本量粗算**（展示统计素养的地方）：转人工率基线 35%，要检出 5pp 差异（α=0.05，power 0.8），每组约需 1,400 会话🔶——按日会话 5,000 的 20% 分流，约 3 天出结果。**低流量期的替代方案**：交替上线（周为单位）+ 前后对比，牺牲严谨换速度，结论降级为"方向性"
- **AI 产品 AB 的特殊护栏**：任何实验组的幻觉抽检不得劣化——质量红线不参与 trade-off

### 4.4 模型选型（评测集就是选型基准）
- **已有资产直接复用**：50 条评测集 + 自动化跑测工具 = 模型回归测试台。换任何模型/供应商，跑一轮四维打分 + 时延 + 成本三张表对比，半天出结论——**选型从"听说哪个好"变成"我测过"**
- 三维权衡框架：质量（四维通过率）× 时延（p95 首 token）× 成本（元/会话）。当前结论：分类 flash / 生成 pro+thinking:false；复评触发条件：新模型发布、供应商调价、质量周报连续 2 周下滑
- **Prompt 与评测集版本化**：prompt 入 git（编排即代码已实现）、评测集随 badcase 持续扩充并打版本号；新 prompt 金丝雀发布（5% 流量 24h 观察护栏指标）后全量

|------|------|-----------|----------|
| 0 验证（现在） | ✅ | Dify Cloud Demo + 评测体系 | 意图体系/PRD/评测规范——**产品逻辑已验证** |

## 六、面试 Q&A 预演（本文档的武器化）

**Q：你这个 Dify Demo 能直接上线吗？**
A：不能，也不该——Demo 是产品验证器，它用一天时间验证了意图体系、对话设计和红线机制成立（94%/红线 0）。真上线走三步：私有化部署解决 SLA 和数据合规、网关层解决集成与安全、埋点监控解决闭环。这三步的方案和 PM 交付物我都写在演进方案里了。

**Q：Dify 的局限你怎么看？**
A：分两类。**当前就要绕的**：Cloud 版限流无 SLA、复杂会话状态管理弱（拍板 A 的两次澄清计数就得靠会话变量勉强实现）；**长期可能触发替换的**：多 Agent 协同、深度定制。但反过来，编排可视化是它给业务侧的最大价值——运营能看懂流程图，这在跨职能协作里值很多钱。工具没有好坏，只有阶段适配。

**Q：性能这么慢怎么办？**
A：先反问一句"哪个环节的慢用户真的在乎"——我的时延预算表里 FAQ 首 token <3s 是硬的，工单摘要 8s 是可接受的。然后按 ROI 排优化：缓存>并行>prompt 瘦身>换模型。平均值不重要，p95 才是投诉来源。

**Q：AB test 你会怎么做第一个实验？**
A：现成的——澄清反问 1 次 vs 2 次。这是 PRD 评审时真实存在过判断分歧的点（我拍了 2 次并设计了缓解措施），正适合数据终审。样本量粗算每组 1,400 会话，护栏是幻觉抽检不得劣化——AI 产品的 AB 和传统 AB 最大的差别就是质量红线不参与 trade-off。
